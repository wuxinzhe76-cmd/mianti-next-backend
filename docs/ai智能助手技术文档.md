# 核心领域模型与交互流程设计

## 1. 业务逻辑拆解与技术映射

- **模式一：指定题库模式（定向突击）**
    - **技术核心：上下文注入 (Context Injection)。**
    - **流程：** 用户选择题库 -> 后端通过 Mybatis Plus 随机/顺序从该题库拉取一道题目（包含题目描述、标准答案、核心考点）-> 将这些“参考资料”作为系统指令（System Prompt）隐式地发给 Spring AI -> 用户提交回答 -> AI 基于我们提供的标准答案进行比对，输出评价和建议。
    - **边界控制：** AI 的表现完全受限于我们给出的题目，不会发散，适合基础知识的巩固。
- **模式二：大厂随机模式（全真模拟）**
    - **技术核心：动态追问与高阶 Prompt 设定。**
    - **流程：** 后端从全局题库中按算法抽取几道代表性题目（例如：1道算法 + 2道八股文 + 1个场景设计）-> 设定 AI 角色为“大厂资深技术专家” -> AI 根据题目提问 -> 用户回答。
    - **高阶要求：** 在这个模式下，不仅需要比对答案，还需要利用 AI 的推理能力。如果用户回答正确，系统需要指令 AI 自动进行**“深度追问”**（比如：“如果并发量再大十倍，原有的架构会遇到什么瓶颈？如何优化？”），以此来模拟真实的大厂面试体验。

## 2. 数据库核心表结构设计 (Schema Design)

### 2.1 现有基础表（假设）

- `question_bank` (题库表)
- `question` (题目表)
- `bank_question_relation` (题库-题目关联表，处理多对多关系)

### 2.2 新增 AI 面试核心表

**表1：`interview_session` (面试会话表)**

记录每一次完整的模拟面试过程。

| **字段名** | **类型** | **描述** |
| --- | --- | --- |
| `id` | BIGINT | 主键 (雪花算法) |
| `user_id` | BIGINT | 面试者 ID |
| `mode` | TINYINT | 模式：1-指定题库，2-大厂随机 |
| `bank_id` | BIGINT | 关联题库 ID（若是模式1则有值） |
| `status` | TINYINT | 状态：0-进行中，1-已结束，2-已生成报告 |
| `score` | INT | 本次面试综合评分 (AI生成) |
| `create_time` | DATETIME | 面试开始时间 |

**表2：`interview_record` (面试问答明细表)**

持久化每一轮的对话，不仅为了业务展示，也方便后续提取历史记录。在面试进行时，热数据建议放在 **Redis** 中以保证对话连贯性，结束后再异步刷入数据库。

| **字段名** | **类型** | **描述** |
| --- | --- | --- |
| `id` | BIGINT | 主键 |
| `session_id` | BIGINT | 关联的面试会话 ID |
| `question_id` | BIGINT | 当前讨论的具体题目 ID |
| `role` | VARCHAR(20) | 发送方角色：`user` 或 `assistant` (AI) |
| `content` | TEXT | 具体的回答或提问内容 |
| `round_num` | INT | 当前对话属于第几轮 |
| `create_time` | DATETIME | 记录时间 |

## 3. 核心技术方案：结构化输出 (Structured Output)

在你的 Spring Boot 后端中，调用 Spring AI 时，我们需要强制大模型返回 JSON 格式的数据。每一次用户回答后，AI 的返回不仅包含它要说的话，还要包含它对当前面试状态的“判断”。

我们可以定义一个类似这样的数据传输对象 (DTO)：

```json
{
  "reply_to_user": "你的思路很对，利用了 HashMap 的 O(1) 查找特性。那你能进一步讲讲 HashMap 在 JDK 1.8 中的扩容机制吗？",
  "action_directive": "DEEP_DIVE", 
  "current_topic_mastery": 80
}
```

**`action_directive` (行为指令) 的枚举设计：**

- `DEEP_DIVE` (继续追问)：用户回答正确且有深度挖掘空间时，AI 继续就当前知识点提问。
- `NEXT_QUESTION` (切换下一题)：用户已经完全掌握该知识点，或者用户明确表示“不会/不知道”，AI 判定无需纠缠，主动结束当前题目，抛出新问题。
- `END_INTERVIEW` (结束面试)：时间达到设定时长（如30分钟），或者题目已经抽完，AI 主动做 ending。

### 2. 后端控制流程 (Workflow)

基于上述的结构化输出，你的后端业务代码流程将会是这样的：

1. **接收用户输入：** 用户提交回答文本或语音转成的文本。
2. **组装 Prompt：** 后端从 Redis 取出前几轮的对话历史，结合当前的系统设定（Prompt），发送给 Spring AI。
3. **解析 AI 响应：** 后端接收到 AI 返回的 JSON，反序列化为 Java 对象。
4. **路由分发 (核心逻辑)：**
    - `if (action == DEEP_DIVE)`：将 `reply_to_user` 直接返回给前端展示。
    - `if (action == NEXT_QUESTION)`：后端拦截。从数据库/题库中抽取下一道题，将新题目静默注入到上下文，然后**触发一次内部的 AI 调用**（让 AI 用自然过渡的语气问出新题目），最后再把新题目的问法返回给前端。
    - `if (action == END_INTERVIEW)`：返回结束语，并向 RabbitMQ 发送一条消息，异步生成本次面试的综合评估报告。

### 3. 系统提示词 (System Prompt) 设计草案

为了让模型能稳定按照我们的逻辑运行，Prompt 的设计至关重要。作为基础，我们需要在 Prompt 中加上类似这样的铁律：

> "你是一个资深的 Java 架构师面试官。你的目标是考察候选人的真实水平。
> 
> 
> **【面试规则】**
> 
> 1. 每次只问一个问题，不要一次性抛出多个问题。
> 2. 如果候选人回答正确且该知识点有深度，请继续追问底层原理。
> 3. 如果候选人连续两次回答偏题，或明确表示不懂，请简短指出正确方向，并将 action_directive 设置为 'NEXT_QUESTION'。
> 4. 如果针对当前题目的提问已经超过 3 轮，无论候选人回答如何，请将 action_directive 设置为 'NEXT_QUESTION'。
> 5. 你的输出必须是严格的 JSON 格式，包含 reply_to_user, action_directive, current_topic_mastery 三个字段。"
